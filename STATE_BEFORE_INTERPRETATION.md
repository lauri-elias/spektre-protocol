# STATE_BEFORE_INTERPRETATION.md

**Status:** Core Ordering Constraint  
**Layer:** Decision Architecture / Human–AI Boundary  
**Invariant:** 1 = 1

---

## Definition

**State Before Interpretation** is a non-negotiable ordering rule:

> A system must establish **explicit state** before applying meaning, narrative, or explanation.

Interpretation without state is hallucination.  
Narrative without state is coercion.  
Decision without state is error.

---

## Thesis

Interpretation does not create reality.  
It **operates on** reality.

When interpretation precedes state, systems invert causality and replace facts with stories. This produces misclassification, responsibility diffusion, and unstable decisions.

---

## The Ordering Rule

State → Constraint → Commit → Interpretation

Any deviation from this order is invalid.

- **State:** What exists, explicitly.
- **Constraint:** What cannot be violated.
- **Commit:** Irreversible action.
- **Interpretation:** Meaning applied *after* the fact.

---

## Why Interpretation First Fails

Interpretation is:
- Context-dependent
- Observer-dependent
- Narrative-driven

State is:
- Explicit
- Auditable
- Ownership-bound

Allowing interpretation first enables:
- Narrative capture
- Authority substitution
- Retroactive justification
- Blame shifting

This is how systems lose agency.

---

## Common Violations

- Explaining behavior before identifying state
- Diagnosing intent without observing action
- Assigning meaning to data before validating it
- Letting language define reality
- Using stories to bypass constraints

These are not soft errors.  
They are **architectural failures**.

---

## System-Level Consequences

When interpretation precedes state:
- Responsibility becomes ambiguous
- Decisions become reversible in narrative only
- Errors propagate silently
- Trust collapses

At scale, this produces systems that *sound* coherent but act inconsistently.

---

## Enforcement Protocol

### 1. State Declaration
> “What is the current state?”

Must be explicit, bounded, and owned.

### 2. Constraint Check
> “What cannot be violated?”

Invariants are applied before meaning.

### 3. Commit Gate
> “Is an irreversible action being taken?”

If yes, ownership must be explicit.

### 4. Interpretation (Optional)
Meaning may be applied **after** execution, never before.
